219 # ---------------------------------------------------------------------------
220 # Pending-ring-free registry — synchronous deferred cleanup at thread exit
221 # ---------------------------------------------------------------------------
222 # When clearProvider(ctx) closes a request broker's ring on the provider
223 # thread, the corresponding poll fn (registered via brokerDispatchLoop's
224 # pollers seq) detects `ring.isClosed()` on its next iteration and needs to
225 # free the shared-memory (ring, slab, pool) triple — but only after a grace
226 # window long enough for any cross-thread sender that snapshotted those
227 # pointers under the previous globalLock state to finish its enqueue.
228 #
229 # Previous design: `asyncSpawn deferredFreeReqRing(...)` — start an async
230 # proc that does `await sleepAsync(50ms)` then frees. Two problems:
231 #
232 # 1. Allocating the sleepAsync Future inside an asyncSpawn started during
233 # `cleanupAllRequestsIdent` runs the refc allocator at a moment where
234 # the thread's gch state is fragile from teardown churn. Observed as a
235 # hard SEGV in rawAlloc on Linux refc + ASAN (PR #13).
236 #
237 # 2. drainAsyncOps only polls chronos for 1ms — the 50ms sleepAsync would
238 # never fire before the processing thread exits, so the buffers either
239 # leak or are freed by an orphaned coroutine racing thread teardown.
240 #
241 # Current design: the poll fn instead records the triple in a thread-local
242 # seq; the processing-thread proc drains the seq AFTER drainAsyncOps via a
243 # single synchronous `sleep(50)` followed by direct free calls. No chronos
244 # involvement in the cleanup path; the grace window applies once for the
245 # whole ctx instead of once per broker.
