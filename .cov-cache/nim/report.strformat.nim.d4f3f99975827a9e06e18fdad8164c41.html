132 # Implementation details
133
134 An expression like `&"{key} is {value:arg} {{z}}"` is transformed into:
135
136 ```nim
137 var temp = newStringOfCap(educatedCapGuess)
138 temp.formatValue(key, "")
139 temp.add(" is ")
140 temp.formatValue(value, arg)
141 temp.add(" {z}")
142 temp
143 ```
144
145 Parts of the string that are enclosed in the curly braces are interpreted
146 as Nim code. To escape a `{` or `}`, double it.
147
148 Within a curly expression, however, `{`, `}`, must be escaped with a backslash.
149
150 To enable evaluating Nim expressions within curlies, colons inside parentheses
151 do not need to be escaped.
152 ]##
153
154 runnableExamples:
155 let x = "hello"
156 assert fmt"""{ "\{(" & x & ")\}" }""" == "{(hello)}"
157 assert fmt"""{{({ x })}}""" == "{(hello)}"
158 assert fmt"""{ $(\{x:1,"world":2\}) }""" == """[("hello", 1), ("world", 2)]"""
159
160 ##[
161 `&` delegates most of the work to an open overloaded set
162 of `formatValue` procs. The required signature for a type `T` that supports
163 formatting is usually `proc formatValue(result: var string; x: T; specifier: string)`.
164
165 The subexpression after the colon
166 (`arg` in `&"{key} is {value:arg} {{z}}"`) is optional. It will be passed as
167 the last argument to `formatValue`. When the colon with the subexpression it is
168 left out, an empty string will be taken instead.
169
170 For strings and numeric types the optional argument is a so-called
171 "standard format specifier".
172
173 # Standard format specifiers for strings, integers and floats
174
175 The general form of a standard format specifier is:
176
177 [[fill]align][sign][#][0][minimumwidth][.precision][type]
178
179 The square brackets `[]` indicate an optional element.
180
181 The optional `align` flag can be one of the following:
182
183 `<`
184 : Forces the field to be left-aligned within the available
185 space. (This is the default for strings.)
186
187 `>`
188 : Forces the field to be right-aligned within the available space.
189 (This is the default for numbers.)
190
191 `^`
192 : Forces the field to be centered within the available space.
193
194 Note that unless a minimum field width is defined, the field width
195 will always be the same size as the data to fill it, so that the alignment
196 option has no meaning in this case.
197
198 The optional `fill` character defines the character to be used to pad
199 the field to the minimum width. The fill character, if present, must be
200 followed by an alignment flag.
201
202 The `sign` option is only valid for numeric types, and can be one of the following:
203
204 ================= ====================================================
205 Sign Meaning
206 ================= ====================================================
207 `+` Indicates that a sign should be used for both
208 positive as well as negative numbers.
209 `-` Indicates that a sign should be used only for
210 negative numbers (this is the default behavior).
211 (space) Indicates that a leading space should be used on
212 positive numbers.
213 ================= ====================================================
214
215 If the `#` character is present, integers use the 'alternate form' for formatting.
216 This means that binary, octal and hexadecimal output will be prefixed
217 with `0b`, `0o` and `0x`, respectively.
218
219 `width` is a decimal integer defining the minimum field width. If not specified,
220 then the field width will be determined by the content.
221
222 If the width field is preceded by a zero (`0`) character, this enables
223 zero-padding.
224
225 The `precision` is a decimal number indicating how many digits should be displayed
226 after the decimal point in a floating point conversion. For non-numeric types the
227 field indicates the maximum field size - in other words, how many characters will
228 be used from the field content. The precision is ignored for integer conversions.
229
230 Finally, the `type` determines how the data should be presented.
231
232 The available integer presentation types are:
233
234 ================= ====================================================
235 Type Result
236 ================= ====================================================
237 `b` Binary. Outputs the number in base 2.
238 `d` Decimal Integer. Outputs the number in base 10.
239 `o` Octal format. Outputs the number in base 8.
240 `x` Hex format. Outputs the number in base 16, using
241 lower-case letters for the digits above 9.
242 `X` Hex format. Outputs the number in base 16, using
243 uppercase letters for the digits above 9.
244 (None) The same as `d`.
245 ================= ====================================================
246
247 The available floating point presentation types are:
248
249 ================= ====================================================
250 Type Result
251 ================= ====================================================
252 `e` Exponent notation. Prints the number in scientific
253 notation using the letter `e` to indicate the
254 exponent.
255 `E` Exponent notation. Same as `e` except it converts
256 the number to uppercase.
257 `f` Fixed point. Displays the number as a fixed-point
258 number.
259 `F` Fixed point. Same as `f` except it converts the
260 number to uppercase.
261 `g` General format. This prints the number as a
262 fixed-point number, unless the number is too
263 large, in which case it switches to `e`
264 exponent notation.
265 `G` General format. Same as `g` except it switches to `E`
266 if the number gets to large.
267 `i` Complex General format. This is only supported for
268 complex numbers, which it prints using the mathematical
269 (RE+IMj) format. The real and imaginary parts are printed
270 using the general format `g` by default, but it is
271 possible to combine this format with one of the other
272 formats (e.g `jf`).
273 (None) Similar to `g`, except that it prints at least one
274 digit after the decimal point.
275 ================= ====================================================

277 # Limitations
278
279 Because of the well defined order how templates and macros are
280 expanded, strformat cannot expand template arguments:
281
282 ```nim
283 template myTemplate(arg: untyped): untyped =
284 echo "arg is: ", arg
285 echo &"--- {arg} ---"
286
287 let x = "abc"
288 myTemplate(x)
289 ```
290
291 First the template `myTemplate` is expanded, where every identifier
292 `arg` is substituted with its argument. The `arg` inside the
293 format string is not seen by this process, because it is part of a
294 quoted string literal. It is not an identifier yet. Then the strformat
295 macro creates the `arg` identifier from the string literal, an
296 identifier that cannot be resolved anymore.
297
298 The workaround for this is to bind the template argument to a new local variable.
299
300 ```nim
301 template myTemplate(arg: untyped): untyped =
302 block:
303 let arg1 {.inject.} = arg
304 echo "arg is: ", arg1
305 echo &"--- {arg1} ---"
306 ```
307
308 The use of `{.inject.}` here is necessary again because of template
309 expansion order and hygienic templates. But since we generally want to
310 keep the hygiene of `myTemplate`, and we do not want `arg1`
311 to be injected into the context where `myTemplate` is expanded,
312 everything is wrapped in a `block`.
313

315 # Future directions
316
317 A curly expression with commas in it like `{x, argA, argB}` could be
318 transformed to `formatValue(result, x, argA, argB)` in order to support
319 formatters that do not need to parse a custom language within a custom
320 language but instead prefer to use Nim's existing syntax. This would also
321 help with readability, since there is only so much you can cram into
322 single letter DSLs.
