1 ## api_cbor_courier — runtime support for the CBOR FFI "buffer courier".
2 ## =====================================================================
3 ## Part C of the CBOR refactoring (doc/CBOR_Refactoring.md §6).
4 ##
5 ## A CBOR-mode `<lib>_call` runs on a foreign caller's thread. Instead of
6 ## decoding CBOR and driving a momentary chronos loop on that foreign
7 ## thread, it becomes a pure courier:
8 ##
9 ## 1. copy the API name into a fixed POD message,
10 ## 2. hand the raw request buffer (by pointer, ownership transferred)
11 ## to the processing thread over a `Channel`,
12 ## 3. block on a per-call response slot until the processing thread
13 ## writes the response back.
14 ##
15 ## The processing thread owns CBOR decode/encode and the provider call.
16 ##
17 ## This module is plain runtime code (NOT codegen) used by the generated
18 ## library runtime in `api_library.nim`. It deliberately contains no Nim
19 ## GC types on the cross-thread message path: `CborCallMsg` is pure POD,
20 ## so a foreign thread can enqueue one with zero GC involvement.
21 ##
22 ## Memory model:
23 ## - `reqBuf` — `allocShared0` by `<lib>_allocBuffer`; ownership moves
24 ## into the `CborCallMsg`; the processing thread frees it exactly once
25 ## after copying the bytes out.
26 ## - `respBuf` — `allocShared0` on the processing thread; ownership
27 ## returns to the `_call` thread via the slot; the foreign caller
28 ## frees it via `<lib>_freeBuffer`.
29 ## - Response slots use a `Lock`+`Cond` (zero OS handles) for the
30 ## blocking handoff — no busy-poll, no per-slot `ThreadSignalPtr`.
31
