1 ## EventBroker
2 ## -------------------
3 ## EventBroker represents a reactive decoupling pattern, that
4 ## allows event-driven development without
5 ## need for direct dependencies in between emitters and listeners.
6 ## Worth considering using it in a single or many emitters to many listeners scenario.
7 ##
8 ## Generates a standalone, type-safe event broker for the declared type.
9 ## The macro exports the value type itself plus a broker companion that manages
10 ## listeners via thread-local storage.
11 ##
12 ## Type definitions:
13 ## - Inline `object` / `ref object` definitions are supported.
14 ## - Native types, aliases, and externally-defined types are also supported.
15 ## In that case, EventBroker will automatically wrap the declared RHS type in
16 ## `distinct` unless you already used `distinct`.
17 ## This keeps event types unique even when multiple brokers share the same
18 ## underlying base type.
19 ##
20 ## Default vs. context aware use:
21 ## Every generated broker is a thread-local global instance. This means EventBroker
22 ## enables decoupled event exchange threadwise.
23 ##
24 ## Sometimes we use brokers inside a context (e.g. within a component that has many
25 ## modules or subsystems). If you instantiate multiple such components in a single
26 ## thread, and each component must have its own listener set for the same EventBroker
27 ## type, you can use context-aware EventBroker.
28 ##
29 ## Context awareness is supported through the `BrokerContext` argument for
30 ## `listen`, `emit`, `dropListener`, and `dropAllListeners`.
31 ## Listener stores are kept separate per broker context.
32 ##
33 ## Default broker context is defined as `DefaultBrokerContext`. If you don't need
34 ## context awareness, you can keep using the interfaces without the context
35 ## argument, which operate on `DefaultBrokerContext`.
36 ##
37 ## Usage:
38 ## Declare your desired event type inside an `EventBroker` macro, add any number of fields.:
39 ## ```nim
40 ## EventBroker:
41 ## type TypeName = object
42 ## field1*: FieldType
43 ## field2*: AnotherFieldType
44 ## ```
45 ##
46 ## After this, you can register async listeners anywhere in your code with
47 ## `TypeName.listen(...)`, which returns a handle to the registered listener.
48 ## Listeners are async procs or lambdas that take a single argument of the event type.
49 ## Any number of listeners can be registered in different modules.
50 ##
51 ## Events can be emitted from anywhere with no direct dependency on the listeners by
52 ## calling `TypeName.emit(...)` with an instance of the event type.
53 ## This will asynchronously notify all registered listeners with the emitted event.
54 ##
55 ## Whenever you no longer need a listener (or your object instance that listen to the event goes out of scope),
56 ## you can remove it from the broker with the handle returned by `listen`.
57 ## This is done by calling `TypeName.dropListener(handle)`.
58 ## Alternatively, you can remove all registered listeners through `TypeName.dropAllListeners()`.
59 ##
60 ##
61 ## Example:
62 ## ```nim
63 ## EventBroker:
64 ## type GreetingEvent = object
65 ## text*: string
66 ##
67 ## let handle = GreetingEvent.listen(
68 ## proc(evt: GreetingEvent): Future[void] {.async.} =
69 ## echo evt.text
70 ## )
71 ## GreetingEvent.emit(text= "hi")
72 ## GreetingEvent.dropListener(handle)
73 ## ```
74
75 ## Example (non-object event type):
76 ## ```nim
77 ## EventBroker:
78 ## type CounterEvent = int # exported as: `distinct int`
79 ##
80 ## discard CounterEvent.listen(
81 ## proc(evt: CounterEvent): Future[void] {.async.} =
82 ## echo int(evt)
83 ## )
84 ## CounterEvent.emit(CounterEvent(42))
85 ## ```
